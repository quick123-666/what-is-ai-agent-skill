# MCP 与 A2A 协议深度解析

## 一、协议层在 Agent 架构中的位置

Agent 系统存在明确的**协议分层**：

| 协议层 | 解决的问题 | 通信主体 |
|--------|-----------|---------|
| **LLM 推理层** | 输入理解、推理、生成 | 用户 ↔ LLM |
| **工具调用层（MCP）** | LLM 如何调用外部工具 | Agent ↔ 工具/数据 |
| **Agent 协作层（A2A）** | Agent 之间如何通信协作 | Agent ↔ Agent |
| **应用层** | 业务逻辑、用户体验 | 用户 ↔ 系统 |

MCP 和 A2A 共同构成 Agent 的**通信基础设施层**，两者互补，缺一不可。

## 二、MCP（Model Context Protocol）深度解析

### 2.1 协议背景

MCP 由 **Anthropic** 于 2024 年底发布，定位为"AI 应用程序的标准协议"。其设计目标与 USB 协议类似——屏蔽底层差异，实现上层互操作。

**核心问题**：在 MCP 出现之前，每个 AI 应用接入外部工具需要独立开发：AI 搜索、AI 数据库、AI 日历、AI 文件系统各自独立，没有统一标准。开发者每接入一个新工具，都要写一套定制化代码。

**MCP 的解决思路**：定义标准化的接口协议，工具提供方实现 MCP Server，应用方（Agent）通过标准 MCP Client 调用，天然互操作。

### 2.2 协议架构

```
┌─────────────────────────────────────────────────┐
│                  Agent (MCP Client)              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│  │ Planner  │→ │ Executor │→ │ Memory   │        │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘        │
│       └─────────────┼─────────────┘              │
│              MCP Client SDK                      │
└───────────────────┼─────────────────────────────┘
                    │ JSON-RPC 2.0 over stdio/HTTP
┌───────────────────┼─────────────────────────────┐
│              MCP Server（工具提供方）             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│  │FileSystem│  │ Database │  │ Calendar │  ...    │
│  └──────────┘  └──────────┘  └──────────┘        │
└─────────────────────────────────────────────────┘
```

**传输层**：JSON-RPC 2.0，支持 stdio（本地进程）和 HTTP（SSE）两种传输方式

**核心原语**：

```json
// Agent → Server：工具调用请求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "filesystem.read_file",
    "arguments": { "path": "/data/report.md" }
  }
}

// Server → Agent：调用结果
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "报告正文..."
  }
}
```

### 2.5 MCP 的局限性

MCP 设计为**同步请求-响应**模式，适合单步工具调用，但对于以下场景存在局限：

1. **长时任务**：需要 Agent 持续跟踪进度的任务（如视频渲染、数据批量处理）
2. **流式输出**：需要实时展示中间结果的场景（如代码补全）
3. **双向通信**：Agent 需要主动向工具推送数据的场景

这些场景需要扩展协议（如 Streaming MCP、Bidirectional MCP），目前是活跃的研究领域。

## 三、A2A（Agent to Agent）深度解析

### 3.1 协议背景

A2A（Agent-to-Agent）由 **Google** 于 2025 年 4 月正式发布，定位为跨平台、跨厂商 Agent 互操作的通信协议。

**核心问题**：企业部署多个 Agent（HR Agent / 财务 Agent / 客服 Agent / 技术 Agent）后，这些 Agent 无法相互通信、无法协作，因为来自不同厂商、使用不同技术栈。

**A2A 的解决思路**：定义 Agent 之间协商任务、交换上下文、汇报结果的标准协议，使不同厂商的 Agent 能够互操作。

### 3.2 协议架构

A2A 的核心设计围绕 **Agent Card**（代理卡）展开：

```
Agent Card（公开接口描述）
├── name: Agent 名称
├── description: Agent 能力描述
├── skills: Agent 擅长的技能列表
├── endpoints: Agent 通信地址
└── capabilities: 支持的协议特性
```

**典型 A2A 工作流**：

```
[Client Agent]                    [Remote Agent]
      │                                │
      │──POST /agents/discover──→     │（返回 Agent Card）
      │                                │
      │──POST /agents/{id}/tasks ──→  │（提交任务）
      │←──202 Accepted─────────────   │（返回 task_id）
      │                                │
      │←GET /agents/{id}/tasks/{tid}──│（轮询任务状态）
      │                                │
      │←─WebSocket 推送进度─────────│（实时推送）
      │                                │
      │←─GET /agents/{id}/tasks/{tid}│（任务完成，获取结果）
      │                                │
```

### 3.3 与 MCP 的互补关系

| 维度 | MCP | A2A |
|------|-----|-----|
| **协议层** | LLM ↔ 工具（工具调用层）| Agent ↔ Agent（协作层）|
| **通信方向** | 单向：Agent 调用工具 | 双向：Agent 相互通信 |
| **粒度** | 原子操作（读文件/查数据库）| 任务级协作（分配/汇报/审核）|
| **类比** | USB 接口（连接外设）| 微信协议（人与人通信）|
| **解决的问题** | AI 不会用工具 | Agent 之间无法协作 |
| **协议状态** | 成熟，稳定生产可用 | 快速增长，企业级落地中 |

**最佳实践**：MCP + A2A 组合使用
- **A2A** 处理 Agent 间的任务分配和状态同步
- **MCP** 处理 Agent 内部对工具的调用

### 3.4 中国生态：微信 A2A 商用里程碑

**2026.06.03**：微信首次向外部 AI 助手开放核心 A2A 接口，这是国内最具影响力的 A2A 商用案例。

## 四、记忆系统：Agent 落地的关键基础设施

### 4.1 为什么记忆是 Agent 的瓶颈

当前 LLM 的 context window 虽然已扩展至 150-200 万 Token，但存在两个根本限制：

1. **成本**：Token 数越多，推理成本线性增长（GPT-4o 128K context 成本是 8K 的 16 倍）
2. **注意力稀释**：超长 context 中，距查询较远的内容被 LLM "遗忘"（Lost in the Middle 问题）

因此，**记忆系统 ≠ 无限制 context**，而是精准检索、高效存储、智能遗忘的工程系统。

### 4.2 MemPalace：本地优先记忆系统标杆

| 维度 | 数据 |
|------|------|
| **GitHub Stars** | 54,763（日增 855 星，2026 年 6 月最大黑马）|
| **召回率** | 96.6% R@5（Top-5 检索召回率）|
| **API Key** | **无需 API Key**，完全本地运行 |
| **向量后端** | ChromaDB / Qdrant / pgvector 等多种支持 |
| **Claude Code 集成** | 深度集成，29 个 MCP 工具 |

**三层记忆架构**：

| 层 | 名称 | 类比 | 说明 |
|----|------|------|------|
| **最浅** | wing | 翅膀/环境 | 跨项目、跨人的全局元信息 |
| **中层** | room | 房间/主题 | 特定项目或话题的上下文 |
| **最深层** | drawer | 抽屉/具体记忆 | 精确到具体事件、对话、决策 |

### 4.3 技术方案对比

| 方案 | 向量库 | 优点 | 缺点 |
|------|-------|------|------|
| **MemPalace** | ChromaDB/Qdrant/pgvector | 三层结构，本地优先 | 配置复杂 |
| **Mem0** | 托管为主 | 托管服务，开箱即用 | 数据上云 |
| **Recursive Mem** | 自研 | 递归记忆压缩 | 闭源 |

## 五、协议安全的四个关键维度

| 安全维度 | MCP 机制 | A2A 机制 |
|---------|---------|---------|
| **认证** | API Key / OAuth 2.0 | Agent Card + TLS + mTLS |
| **授权** | 工具粒度权限控制 | 技能级别访问控制 |
| **审计** | 完整调用日志 | 任务级操作记录 |
| **传输安全** | stdio（本地）/ HTTPS（SSE）| HTTPS + WebSocket |

**企业部署建议**：
- MCP Server 部署在内网，通过 API 网关暴露
- A2A Agent 间通信使用 mTLS 双向认证
- 日志统一接入 SIEM 系统（如 Splunk / Elastic）

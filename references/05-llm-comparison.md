# 大模型三角深度对比（2026 年 6 月）

## 一、技术评测基准详解

### 1.1 SWE-bench：编程能力的行业基准

SWE-bench（Software Engineering Benchmark）是评估 LLM 编程能力的权威基准，测试集来自真实 GitHub Issue，要求模型：
1. 理解 Issue 描述
2. 定位相关代码文件
3. 生成正确的修复补丁（pass rate = 测试通过率）

**2026 年 6 月 SWE-bench 排行榜**：

| 排名 | 模型 | 得分 | 备注 |
|------|------|------|------|
| **1** | Claude Opus 4.7 | **87.6%** | 当前编程能力最强 |
| 2 | GPT-5.6（内测）| ~85% | 预计发布后接近 Opus 水平 |
| 3 | Gemini 3.5 Ultra | 78.3% | Google 旗舰，多模态强 |
| 4 | Kimi K2.6 | 72.1% | 国产第一，长上下文优势 |
| 5 | DeepSeek V4 | 68.5% | 开源版本可供微调 |

## 四、上下文窗口的技术意义

### 4.1 为什么 Token 数是 Agent 的关键指标

LLM 的 context window（上下文窗口）限制了在单次推理中能输入的最大 Token 数。这对 Agent 的影响是根本性的：

| Context 长度 | 实际意义 | Agent 能力 |
|-------------|---------|-----------|
| **8K Token** | ~6000 中文 | 单次对话，跨天任务完全断片 |
| **128K Token** | ~10 万中文 | 能处理一篇论文，一段代码库 |
| **1M Token** | ~75 万中文 | 能处理一部长篇小说，一个中大型代码库 |
| **2M Token** | ~150 万中文 | 能处理《大英百科全书》全书，跨天任务不断片 |

### 4.2 Lost in the Middle：超长 context 的隐藏问题

即便模型支持超长 context，仍然存在 **"中间遗忘"（Lost in the Middle）** 问题：距查询较远的内容被 LLM "遗忘"，检索精度下降。

**解决方案**：
1. **注意力机制优化**：改进的 attention pattern（如 Longformer / Ring Attention）
2. **混合检索**：先用向量检索定位相关段落，再注入 context（如 RAG + long context 组合）
3. **记忆系统**：用 MemPalace 等记忆系统替代纯 context
